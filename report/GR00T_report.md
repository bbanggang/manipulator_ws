### 3.2 근본 원인 규명 — **covariate shift (누적 오차)** (2026-07-20)

v4(vis2)가 loss 0.009로 v1/v2와 동등한데도 증상이 재현되자, "vision freeze가 원인"이라는
가설이 **반증**됐다. 이후 서버에 관측을 직접 주입하는 오프라인 프로브 4종으로 원인을 좁혔다.

#### 3.2.1 요인 배제 (무엇이 원인이 **아닌가**)

| 검증 | 방법 | 결과 |
|---|---|---|
| 모델의 시각 그라운딩 | `probe_swap_images.py` — state 고정, **영상만 좌↔우 교체** | ✅ **존재**. 영상 효과 18.9° vs state 효과 1.9° (10배, 4쌍 일관) |
| 학습 분포에서의 추종 | `probe_training_obs.py` — 학습 프레임 주입 vs GT 액션 비교 | ✅ **정확**. 중앙 큐브 ep3/8/13/18 전 구간 GT 추종, 좌측 드리프트 없음 |
| 시작 자세 | `goto_home.py` 후 state를 학습 50ep 분포와 z-score 비교 | ❌ 원인 아님 (전 관절 \|z\|<0.4) |
| 조명/밝기 | `probe_brightness_causal.py` — 학습 프레임을 배포 밝기로 감광 | ❌ 원인 아님 (평균 효과 **+0.1°**) |
| 카메라 구도 이동 | 밝기 정규화 후 지형지물 비교(매트 좌측경계·흰면적) | ❌ 변화 없음 (경계 72 vs 74~80px) |
| 실행/클램프 버그 | `actions.csv`(명령) vs `chunks.csv`(결과) 대조 | ❌ 아님 — **모델이 직접** 좌측을 명령 (state -5.6 → 명령 -8.3) |
| RGB/BGR 뒤바뀜 | 로깅 경로 역추적(저장 영상에서 큐브가 빨강) | ❌ 아님 |

#### 3.2.2 원인 확정 — wrist 뷰 오프셋과 명령의 **역상관**

배포 실행(HORIZON=16) 전 구간에서 **wrist 뷰 속 큐브의 좌우 오프셋**과 **다음 청크의 pan 변화**를
대조한 결과:

| chunk | pan | wrist 큐브 오프셋 | 다음 Δpan | 판정 |
|---|---|---|---|---|
| 7 | -4.5 | +58px | -2.0 | ✗ 반대 |
| 9 | -10.1 | +140px | -3.2 | ✗ 반대 |
| 10 | -13.3 | +187px | -3.8 | ✗ 반대 |
| 12 | -22.1 | +267px | -5.1 | ✗ 반대 |

> **상관계수 = -0.86.** 큐브가 오른쪽으로 벗어날수록 모델은 **더 왼쪽**을 명령하며,
> **오차가 커질수록 이탈 속도도 빨라진다**(가속 발산).

학습 데이터에서 시연자는 큐브를 wrist 중심 근처(≈+59px)에 항상 유지했다. 배포에서 관측된
**+267px는 학습 분포에 존재하지 않는 상태**다. 즉 모델은 "어긋났을 때 되돌아오는 법"을
배운 적이 없고, 그 영역에서의 출력은 사실상 임의(=지배적 최좌측 모드로 낙하)다.

#### 3.2.3 결론과 재해석

**실패 원인은 모델·학습설정이 아니라 데이터의 성격이다** — 50ep 전부가 시연자가 정확히
수행한 **깨끗한 성공 시연**이라 교정(recovery) 예시가 0건이다. 청크 단위 불연속 실행 때문에
작은 이탈은 필연적으로 발생하는데, 그 순간 학습에 없는 상태로 진입해 발산한다.
행동복제의 고전적 실패인 **covariate shift / compounding error**([DAgger, Ross et al. 2011,
arXiv:1011.0686](https://arxiv.org/abs/1011.0686))에 정확히 부합한다.

이 관점에서 v1~v4의 모든 관측이 **하나의 원인**으로 통합 설명된다:

| 기존 해석 | 재해석 |
|---|---|
| v1 "idle 어트랙터" | 이탈 후 학습 분포 밖 → 정지 모드로 낙하 |
| v2 "모드 고정(pan -77)" | 이탈 후 지배적 최좌측 모드로 낙하 |
| v3 "학습부족" | (이건 실제로 학습부족이 맞음 — v4로 해소 확인) |
| v4 좌측 드리프트 | 동일 원인의 직접 관측 (역상관 -0.86) |
| 시행마다 실패 양상 상이 | 초기 이탈 방향·크기에 따라 낙하하는 모드가 달라짐 |

부수적으로 **HORIZON 8→16이 그리퍼 개방을 복구**시켰다(3.2→**49.1**, 학습 최대 49와 일치).
동기식 클라이언트는 8스텝(160ms) 실행 후 추론(~300ms) 동안 정지해 **약 65% 시간을 멈춰 있는데**,
16스텝으로 늘려 정지 비율을 낮추자 이탈이 줄어 궤적 후반(파지 단계)까지 도달했다.
이는 원인 진단과 정합하며, **실행 리듬도 이탈량에 기여**함을 보여준다.

### 3.3 진단 방법론 (재사용 자산)

- **open-loop 평가**(`eval_policy.py`): 학습 데이터 재생으로 모델/배포 문제 분리 — v1이
  GT를 완벽 추종함을 확인해 "모델은 정상, 배포 조건이 문제"로 조기 확정
- **관측 비교 실험**: 라이브 관측 vs 학습 첫프레임을 동일 zmq 서버에 주입 → 이미지 통계·
  state 일치, **둘 다 출발 명령 생성** 확인 (전처리 무결 입증)
- **클라이언트 계측**: rerun(카메라·state·action) + CSV(청크별 state/추론지연, 스텝별 액션)
  + 정책 시점 video.mp4 — "movement인지 진동인지 주차인지"를 수치로 판정
- **캘리브레이션 검증**: 7/10 재캘리브레이션 전후 좌표계 비교 — 신 캘리브레이션이 학습
  좌표계와 일치(구본은 wrist_roll -54° 어긋남 = 혼 슬립을 신본이 흡수). 홈 자세 손배치
  오차가 ±10°에 달해 **`goto_home.py`(학습 평균 자세 ±2° 복귀)를 측정 프로토콜에 도입**

### 3.2 인프라 (원격 추론)

## 5. 현재 상태와 다음

**실기 T1은 정식 10회 측정에 진입하지 못했다** — §3.2에서 원인이 데이터 성격(교정 시연 부재)으로
확정되어, 같은 데이터로 하이퍼파라미터만 바꾸는 반복은 더 이상 정보 가치가 없다고 판단.

### 5.2 sim-to-real로 넘어가는 근거

이번 규명은 **sim 단계의 필요성을 강화한다**: 교정 데이터는 실기에서 손으로 모으기 번거롭고
(어긋난 자세를 일일이 만들어야 함) 재현성도 낮은데, 시뮬레이터에서는 **임의 초기자세로부터
자동·대량 생성**이 가능하다. 도메인 랜덤화(DR)까지 얹으면 상태 분포를 훨씬 넓게 덮을 수 있다.
→ [model_markdown/sim2real/04_GR00T_sim.md](../model_markdown/sim2real/04_GR00T_sim.md)

- T2는 v1 학습만 완료(loss 0.0105)한 상태로 보류 — 동일 원인이 예상되므로 실기 반복 대신
  sim 경로에서 처방 검증 후 재개

### 5.3 진단 도구 (본 규명에서 신규 작성, 재사용 가능)

`setup/gr00t/`에 추가 — π0/GR00T뿐 아니라 **ACT/SmolVLA 진단에도 이식 가능**:
- `probe_training_obs.py` — 학습 관측 주입 후 GT 액션과 대조 (state 분기 교란 통제 포함)
- `probe_swap_images.py` — state 고정·영상만 교체하는 통제 실험 (시각 사용 여부 판정)
- `probe_deploy_frame.py` — 배포 프레임 주입으로 실패 오프라인 재현 + 요인 제거
