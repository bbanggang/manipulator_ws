# 04. GR00T N1.5 (3B) — Zero-shot (파이프라인 검증 목적)

> **역할**: "대형 파운데이션을 16GB에서 LoRA로 길들이면 SmolVLA를 이기는가" 검증 +
> 향후 Isaac Sim 합성데이터 경로(LeIsaac)와 직결. SO-100/101은 사전학습에 미포함(OOD)이라
> **zero-shot 성능은 최저 예상 — 파이프라인 검증이 주목적**.
> 버전: **N1.5로 시작** (레퍼런스 최다), 여유 시 N1.7 추가 실험.

## 모델 개요

| 항목 | 내용 |
|---|---|
| 파라미터 | 3B |
| 사전학습 체크포인트 | ✅ `nvidia/GR00T-N1.5-3B` |
| 사전학습 데이터 | Data Pyramid (실로봇+인간영상+합성) — **SO-100/101 미포함** → `new_embodiment`로 FT 필수. 단 60ep 미만 FT로 SO-ARM101 성공 사례 다수 |
| 아키텍처 | Eagle-2 VLM + DiT 듀얼시스템 |
| 학습 입력 | 이미지 + state + 언어 (**GR00T-LeRobot** = LeRobot + `modality.json`) |
| 추론 구조 | **inference server + client** (LeRobot client 연결), embodiment tag 필요 |
| 출력 | DiT 기반 action chunk (horizon ≤16 관찰됨) |
| 5070 Ti 16GB | ⚠️ FT는 LoRA + `--no-tune_diffusion_model`로 가능 (full은 ~25GB). 커뮤니티 4080 16GB LoRA(rank 16~32, batch ≤16) 성공 사례 복수 |
| RTX 5090 32GB | ✅ Full FT 가능 (~25GB < 32GB) → "16GB LoRA vs 32GB full" 비교 실험 포인트 |

## 데이터 요구사항 (향후 FT 대비)

- **포맷**: GR00T-LeRobot = LeRobotDataset + `meta/modality.json` (state/action 슬라이싱, 카메라 key 매핑)
- SO-100 듀얼캠 예제의 `so100_dualcam__modality.json`을 데이터셋 `meta/`에 복사하면 "GR00T-호환" 완성
- embodiment tag: `new_embodiment`
- ⚠️ **주의 1**: 학습/추론 시 **카메라 이름이 수집 당시와 정확히 일치**해야 함 (실패 원인 1순위)
  - 우리 네이밍은 `top`/`wrist` — `so100_dualcam` 기대 key와 다르면 `modality.json`에서 매핑 정의
- ⚠️ **주의 2**: 버전 민감 — GR00T 최신(N1.7)은 v3→v2 변환 스크립트를 별도 제공할 정도. 요구 버전 선확인

## 환경 (별도 격리 필수)

- `envs/gr00t/` uv 프로젝트 분리 (flash-attn 등 무거운 의존성)
- **설치는 게이트 예외**: 앞 모델(D4~D6) 진행 중 백그라운드 병행 허용
  - 사전컴파일 휠이면 짧고, 소스 빌드면 매우 김 — 미리 착수 (7/11 토 버퍼 활용)
- data-config: `so100_dualcam`

## 향후 확장 메모

- **Isaac Sim 합성데이터 co-training**: LeIsaac이 SO-101 teleop → LeRobot 변환 → GR00T FT → Isaac Lab 평가 전 과정 제공.
  실데이터 + 합성 혼합 co-training이 실데이터 단독 대비 평균 +38% (arXiv:2503.24361). 순수 합성 only는 reality gap으로 위험.
- **양팔 확장 시**: `so100_dualcam__modality.json`은 단일팔 전용 → 양팔(≈12-DOF)은 state/action 슬라이싱 직접 정의 필요 (추가 공수 포인트)
